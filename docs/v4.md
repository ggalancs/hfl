# HFL V4 — features beyond Ollama

V4 turns HFL into a HuggingFace-Hub-native LLM server. Every feature
in this guide is something Ollama either doesn't have, or can't have
without rewriting around its curated registry.

---

| Feature | REST | CLI |

| Hub discovery | `GET /api/discover` | `hfl discover` |

## Hub discovery — `hfl discover`

Filter the live HF Hub by family, quantisation, multimodal, license,
likes, downloads. Cached on disk for 5 minutes (override with
`--refresh`).

```bash
# Top 20 GGUF Qwens with at least 100 likes
hfl discover qwen --quant gguf --min-likes 100

# Multimodal Llama variants from the Hub
hfl discover --family llama --multimodal

# Hit the cache; --refresh forces a network round-trip
hfl discover qwen --refresh
```

The `local` column marks models you already have in your registry.

## HW-aware recommendation — `hfl recommend`

Probe the host (RAM, VRAM, MLX availability) and rank Hub models by
hardware fit + capability fit + popularity + recency. Models that
overflow the budget are excluded outright — no advice that won't run.

```bash
hfl recommend --task chat --top 5
hfl recommend --task code --family qwen
```

Output explains *why* each pick — "fits comfortably (7.8/24.0 GB)",
"matches task 'chat' (tag 'instruct')", etc.

---

## Smart pull — `hfl pull-smart`

Give it a base repo, it finds the best community variant for your
hardware. On Apple Silicon: `mlx-community/...-4bit`. On CUDA 24GB:
`bartowski/...-GGUF/Q5_K_M`. CPU-only: smallest quant that fits.

```bash
hfl pull-smart meta-llama/Llama-3.1-8B-Instruct
hfl pull-smart Qwen/Qwen2.5-Coder-32B-Instruct --max-vram-gb 16
```

Prints the resolved plan first (target_repo_id, quant, estimated
VRAM, why it picked that), then forwards to the regular pull.

## Verify — `hfl verify`

Five sanity checks in seconds, before you commit a freshly-pulled
model to production:

- tokenizer round-trip
- chat-template render
- 1-token smoke generation
- tool-parser
- embedding dimension (when applicable)

```bash
hfl verify qwen-coder-7b
```

Exits non-zero if any check fails. Useful in CI hooks before
restarting the inference fleet.

## Benchmark — `hfl bench`

TTFT (time-to-first-token) + tokens/sec on golden prompts of three
lengths (16, 256, 2048 chars), with p50/p95 percentiles.

```bash
hfl bench qwen-coder-7b
hfl bench qwen-coder-7b --runs 5 --max-tokens 128 --lengths 64,1024
```

The TTFT column shows `—` when the engine has no `generate_stream`
hook (a no-stream engine cannot honestly measure first-token
latency — we report `None` rather than aliasing it to total time).

## LoRA hot-swap — `hfl lora`

Apply / remove LoRA adapters without reloading base weights.
Multiple adapters stack at fractional scales (engine-supported).

```bash
hfl lora apply qwen-7b --path adapters/code.safetensors --scale 0.7 --name code
hfl lora apply qwen-7b --path adapters/legal.safetensors --scale 0.3 --name legal
hfl lora list qwen-7b
hfl lora remove qwen-7b --id <adapter-uuid>
```

## KV snapshots — `hfl snapshot`

Save the KV cache after loading a long system prompt; restore on
the next server start to skip the prefill.

```bash
hfl snapshot save qwen-7b --name warm-prod
hfl snapshot list
hfl snapshot load qwen-7b --name warm-prod
hfl snapshot delete --name warm-prod
```

Snapshots are model-bound — loading a snapshot from model A into
model B is rejected with a 400. Format version is stamped on the
sidecar; a future llama-cpp-python upgrade that re-shapes KV
tensors will reject old snapshots cleanly rather than corrupting
memory.

## Compliance dashboard — `hfl compliance-dashboard`

License risk overview of the local registry: bucket counts
(permissive / conditional / non-commercial / restricted / unknown),
gated repos pending HF_TOKEN, missing licenses, EU AI Act warnings
for restricted licenses.

```bash
hfl compliance-dashboard
```

Different from `compliance-report` (which generates a per-model
Markdown audit trail).

## Speculative decoding draft — `hfl draft-recommend`

Auto-pick a smaller sibling on the Hub for speculative decoding.
Falls back to the canonical small reference (Llama-3.2-1B,
Qwen2.5-1.5B, gemma-3-270m, Mistral-7B) when no closer fit exists.

```bash
hfl draft-recommend meta-llama/Llama-3.1-70B-Instruct
hfl draft-recommend Qwen/Qwen3-32B --max-ratio 0.1
```

Use the resolved repo with `DRAFT` in a Modelfile or pass it as
`draft_model_path` at engine load. **Speedup is workload-dependent**;
benchmark with your prompt mix before committing — measured 1.33×
on Qwen3-14B + Qwen3-0.6B for a structured-JSON prompt, but ~0.7×
for free-form prose where the draft predictions get rejected.

## WebSocket chat — `WS /ws/chat`

Persistent connection with frame-level cancellation. The HTTP
streaming endpoints close TCP on cancel; WebSocket sends an
explicit `cancel` frame and keeps the connection open for the
next prompt.

Frame grammar (JSON):

```
client → server:
  {"type": "chat", "model": "...", "messages": [...], "options": {...}?}
  {"type": "cancel"}
  {"type": "ping"}

server → client:
  {"type": "ready", "model": "..."}
  {"type": "token", "delta": "..."}
  {"type": "done", "tokens": N}
  {"type": "cancelled", "tokens": N}
  {"type": "error", "message": "..."}
  {"type": "pong"}
```

Authentication: when `HFL_API_KEY` is configured, the upgrade must
carry it via `?api_key=...`, `Authorization: Bearer ...`, or
`X-API-Key`. Origin allow-list (`HFL_ORIGINS`) applies too.

## Push to HF Hub — `POST /api/push`

Upload a registered local model to the HF Hub.

```bash
curl -X POST http://localhost:11434/api/push \
  -H 'Content-Type: application/json' \
  -d '{
    "model": "my-finetuned-7b",
    "destination": "my-org/my-finetuned-7b",
    "private": true,
    "stream": true
  }'
```

NDJSON progress mirrors `/api/pull`. Auth chain: explicit `token`
in the body → `config.hf_token` (which reads `HF_TOKEN` env at
boot) → unauthenticated (fine for public repos, fails for private).

## OpenAI Responses API — `POST /v1/responses`

The 2025+ OpenAI surface — used by the new `client.responses.create()`
SDK. HFL maps it onto the existing chat-completion path.

Supported on the server side: string/list `input`, `instructions`,
`tools`, `reasoning.effort`, `response_format`, both `stream=true`
(SSE with `response.created` / `response.output_text.delta` /
`response.completed` / `[DONE]`) and `stream=false` envelopes.

Not stateful — every request is self-contained, just like Ollama's
own `/v1/responses` limitation.

## Environment variables added in V4

See `docs/env-vars.md` for the full table. V4-relevant additions:

- `HFL_NUM_PARALLEL` / `OLLAMA_NUM_PARALLEL` — dispatcher slot count.
- `HFL_MAX_LOADED_MODELS` / `OLLAMA_MAX_LOADED_MODELS` — multi-model pool.
- `HFL_KEEP_ALIVE` / `OLLAMA_KEEP_ALIVE` — global default keep-alive.
- `HFL_KV_CACHE_TYPE` — KV cache dtype (f16/q8_0/q4_0).
- `HFL_FLASH_ATTENTION` — global flash-attn toggle.
- `HFL_ORIGINS` / `OLLAMA_ORIGINS` — CORS allow-list.
- `HFL_LLM_LIBRARY` / `OLLAMA_LLM_LIBRARY` — pin auto-selection.
- `HFL_DEBUG` / `OLLAMA_DEBUG` — DEBUG-level logging.

Drop-in replacement of an Ollama install: most `OLLAMA_*` variables
are accepted as fallbacks when the explicit `HFL_*` variant is not
set.
